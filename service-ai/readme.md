# 🪽 FeatherRAG 

## 一些思考

所以把RAG剥离出来，并且用LangChain重构

框架应该专注于接口抽象和编排，而MCP，RAG都应该看作是框架接口的具体实现，比如可以作为Tool的实现，也可以作为一个编排的节点

实实在在感受到了使用现有框架重构的好处，也体会到了langchain的生态强大，虽然它的API非常混乱，层次不清，但是很明显它的先发优势让其得到了最高支持优先级，且由于统一接口，数据的流转变得非常简单

不过这让底层实现成为了黑盒，有时候反而阻碍理解，比如在高层次的封装下你不知道向量数据库到底发生了什么

所以这也是手搓框架的意义吧，如果只理解上层封装过的接口，遇到一些小问题就会完全不知所措

## 核心流程

1. load 从各种格式转化为结构化文本 
	- 原有实现：markitdown
	- 优化：langchain-unstructured package
2. chunk 文档分块
	- 原有实现： 手写markdown标题分块算法
	- 优化：langchain-text-split 
    - 未来继续探索优化分块策略，根据文件类型
3. embed 嵌入
	- 增加bm25算法的稀疏向量嵌入 
4. retriver
	- 混合检索 稀疏/稠密向量 RRF
	- 查询翻译 MQE，Hyde，进一步支持step-back-prompt
	- 查询路由 ？只有一个数据库的话没有必要
	- 进阶技术 ？可以看情况尝试
5. 评估
	- 为项目提供数据支撑，量化数据的可信度更高
6. 封装
	- 做成一个知识库问答系统
